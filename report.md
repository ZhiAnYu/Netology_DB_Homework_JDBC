## 1. Цель работы
Создать Spring Boot приложение, работающее с базой данных PostgreSQL, запущенной в Docker-контейнере. 
Реализовать слой доступа к данным (DAO) с использованием `NamedParameterJdbcTemplate` и именованных параметров, 
вынести SQL-запросы в отдельные файлы в папке `resources`, разработать REST-контроллер для обработки HTTP GET-запросов и корректно обрабатывать случаи отсутствия данных в базе.

## 2. Использованные технологии и инструменты
- **ОС:** Windows 11 с Ubuntu WSL (Windows Subsystem for Linux) и Docker Engine
- **Среда разработки:** IntelliJ IDEA (Community)
- **Язык программирования:** Java 17
- **Сборщик проектов:** Apache Maven
- **Фреймворк:** Spring Boot 3.x
  - `spring-boot-starter-web` — для REST API и Tomcat
  - `spring-boot-starter-jdbc` — для работы с JDBC и `NamedParameterJdbcTemplate`
- **База данных:** PostgreSQL 16 (запущен в Docker-контейнере)
- **Драйвер БД:** `org.postgresql:postgresql`
- **Контейнеризация:** Docker Engine
- **Инструменты управления БД:** DBeaver (для визуального проектирования схемы, выполнения SQL-скриптов и отладки запросов)
- **Инструменты тестирования API:** Postman (для отправки HTTP-запросов, проверки параметров и валидации ответов контроллера), `curl`
- **Вспомогательные инструменты:** Qwen (AI-ассистент для генерации boilerplate-кода, отладки синтаксиса SQL/Java, рефакторинга и структурирования итогового отчета)

## 3. Выполненные шаги

### 3.1. Генерация Spring Boot проекта
Проект создан через Spring Initializr с необходимыми зависимостями: `Spring Web`, `JDBC API` и `PostgreSQL Driver`. Структура проекта организована в пакете `netology.HomeworkJDBC`.

### 3.2. Настройка инфраструктуры БД

Запущен Docker-контейнер с PostgreSQL командой:
```bash
docker run --name postgres \
  -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 \
  -v pg-data:/var/lib/postgresql/data \
  -d postgres:16
```

Настроено подключение в `application.properties`:

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/postgres
spring.datasource.username=postgres
spring.datasource.password=postgres
spring.sql.init.mode=always
```

### 3.3. Создание SQL-скриптов
- **`schema.sql.bac`** — скрипт автоматического создания схемы `netology_homework`, таблиц `customers` и `orders` с внешним ключом, а также наполнение тестовыми данными.
- **`query.sql`** — параметризованный запрос с JOIN, использующий именованный параметр `:name`, функцию `LOWER()` для нечувствительности к регистру и `LIMIT 1` для защиты от `IncorrectResultSizeDataAccessException`.

### 3.4. Реализация слоя Repository
Создан класс `OrderRepository` с аннотацией `@Repository`:
- Внедрение `NamedParameterJdbcTemplate` через конструктор.
- Загрузка SQL-скрипта из `resources` в строковую переменную при инициализации бина через метод `read()` с использованием `ClassPathResource` и `BufferedReader`.
- Реализация метода `getProductName(String name)` с использованием `Map.of("name", name)` для передачи параметров и явного `RowMapper` в виде лямбда-выражения `(rs, rowNum) -> rs.getString("product_name")`.

### 3.5. Реализация REST-контроллера
Создан `OrderController` с аннотацией `@RestController`:
- Маппинг GET-запросов на эндпоинт `/products/fetch-product`.
- Получение параметра `name` через `@RequestParam`.
- Обработка `EmptyResultDataAccessException` с возвратом HTTP 404 Not Found и понятным сообщением.
- Общая обработка исключений с возвратом HTTP 500 Internal Server Error.

## 4. Проблемы и их решения

1. **Spring Boot не выполняет `schema.sql.bac` при старте**
   - **Причина:** По умолчанию Spring Boot запускает скрипты инициализации схемы только для встроенных баз данных (например, H2).
   - **Решение:** Добавлено свойство `spring.sql.init.mode=always` в файл `application.properties` для принудительного выполнения скриптов при любом типе БД.

2. **`IncorrectResultSizeDataAccessException` при наличии нескольких заказов**
   - **Причина:** Метод `queryForObject` в Spring JDBC строго ожидает ровно одну строку результата. Если у клиента несколько заказов, возникает ошибка.
   - **Решение:** Добавлено ограничение `LIMIT 1` в конец SQL-запроса в файле `get_product.sql`.

3. **`EmptyResultDataAccessException` при запросе несуществующего клиента**
   - **Причина:** Метод `queryForObject` выбрасывает исключение, если запрос не вернул ни одной строки, вместо того чтобы вернуть `null`.
   - **Решение:** Вызов метода репозитория в контроллере обернут в блок `try-catch`, который перехватывает это исключение и возвращает клиенту корректный HTTP-ответ 404 Not Found.

4. **Поиск по имени клиента не работает без учета регистра**
   - **Причина:** В PostgreSQL оператор `=` выполняет чувствительное к регистру сравнение строк.
   - **Решение:** В условии `WHERE` запроса к обеим сторонам сравнения применена функция `LOWER()`, что делает поиск нечувствительным к регистру (например, "Alexey" и "alexey" будут найдены).

5. **`Connection refused` при запуске приложения**
   - **Причина:** Docker-контейнер с PostgreSQL не был запущен, либо порт 5432 был занят локально установленной базой данных Ubuntu.
   - **Решение:** Проверка состояния контейнера через `docker ps`. При конфликте портов локальная служба PostgreSQL останавливалась командой `sudo systemctl stop postgresql`.

6. **Ошибки синтаксиса SQL при выполнении `schema.sql.bac`**
   - **Причина:** Опечатки в SQL-скрипте, такие как лишняя запятая перед закрывающей скобкой в операторе `INSERT` или отсутствие одинарных кавычек вокруг значений типа `DATE` или `VARCHAR` (например, `2023-10-25` вместо `'2023-10-25'`).
   - **Решение:** Внимательная проверка синтаксиса `schema.sql.bac`. Удаление лишних запятых и заключение всех строковых и датных значений в одинарные кавычки.

## 5. Чек-лист самопроверки
- Spring Boot приложение запускается без ошибок и подключается к PostgreSQL в Docker
- `schema.sql.bac` автоматически выполняется при старте, создавая таблицы и тестовые данные
- SQL-запрос вынесен в отдельный файл `query.sql` и читается через `ClassPathResource`
- В запросе используется именованный параметр `:name`
- Репозиторий помечен аннотацией `@Repository` и использует `NamedParameterJdbcTemplate`
- В методе `getProductName` используется `Map.of` для параметров и `RowMapper` через лямбду
- Контроллер обрабатывает GET-запросы по пути `/products/fetch-product?name=...`
- При существующем клиенте возвращается название продукта (HTTP 200)
- При несуществующем клиенте возвращается HTTP 404 с понятным сообщением
- Поиск работает без учета регистра (Oleg = oleg = OLEG)

## 6. Полученные результаты

**Работающий REST API:**
- `GET /products/fetch-product?name=Oleg` → `Orbit` (HTTP 200)
- `GET /products/fetch-product?name=oleg` → `Orbit` (HTTP 200, нечувствительность к регистру)
- `GET /products/fetch-product?name=Ivan` → `Продукт для пользователя 'Ivan' не найден` (HTTP 404)

**Архитектурные решения:**
- Разделение бизнес-логики и SQL-кода: запросы хранятся в отдельных `.sql` файлах, что облегчает их редактирование и тестирование в IDE.
- Безопасность от SQL-инъекций: использование `NamedParameterJdbcTemplate` с именованными параметрами вместо конкатенации строк.
- Корректная обработка ошибок: HTTP 404 для отсутствующих данных вместо "падения" приложения с 500 ошибкой.
- Автоматическая инициализация БД: схема и тестовые данные создаются при каждом запуске благодаря `schema.sql.bac`.

## 7. Вывод
В ходе выполнения задачи были закреплены ключевые навыки работы со Spring JDBC:
- Организация слоя доступа к данным (DAO) с использованием паттерна Repository.
- Работа с именованными параметрами через `NamedParameterJdbcTemplate` и `MapSqlParameterSource`.
- Чтение SQL-скриптов из `classpath` ресурсов с помощью `ClassPathResource`.
- Использование явного `RowMapper` для преобразования результатов запроса в Java-объекты.
- Интеграция с реляционной базой данных PostgreSQL, запущенной в Docker-контейнере в среде Ubuntu WSL2.
- Автоматическая инициализация схемы через `schema.sql.bac` и настройка через `application.properties`.
- Построение RESTful API с корректной обработкой исключений и HTTP-статусов.

Реализованное приложение полностью соответствует требованиям задания, готово к промышленной эксплуатации и может быть расширено дополнительным функционалом (пагинация, кэширование, транзакционность) без изменения базовой архитектуры.
